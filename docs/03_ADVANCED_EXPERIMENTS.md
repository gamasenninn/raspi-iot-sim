## 実験6: REST API テスト (`demo_rest_test.py`)

### 目的
Pi VM 内で動作する Flask REST API の全エンドポイントを網羅テスト。

### テスト項目と結果

| # | テスト | メソッド | 結果 |
|---|--------|---------|------|
| 1 | ヘルスチェック | GET /api/health | 200 OK, devices:15 |
| 2 | 全デバイス一覧 | GET /api/devices | 200 OK, 15デバイス |
| 3 | 特定デバイス | GET /api/devices/servo | 200 OK, actions含む |
| 4 | LED ON | POST /api/devices/led_red/action | 200 OK, on:true |
| 5 | サーボ位置設定 | POST ... {set, 0.75} | 200 OK, value:0.75 |
| 6 | RGB LED色設定 | POST ... {set, "0.2,0.8,0.3"} | 200 OK |
| 7 | モーター前進 | POST ... {forward} | 200 OK, speed:1.0 |
| 8 | 存在しないデバイス | GET /api/devices/nonexistent | 404 |
| 9 | 不正なアクション | POST ... {fly} | 400, available actions表示 |

### 知見
- REST API は Pi VM 内の `localhost:5000` で正常動作
- エラーレスポンスが適切（404 に使用可能アクション一覧を含む）
- `requests` ライブラリで Pi VM 内から自分自身のAPIを叩ける
- **現在の制限**: Pi VM の port 5000 はQEMUの外にフォワードされていないため、外部から直接アクセスできない。外部アクセスにはPI-CIの`hostfwd`設定変更が必要

## 実験7: スマートホーム自律制御 (`demo_smart_home.py`)

### 目的
複数センサー値に基づくマルチデバイスの自律制御ロジック検証。

### 制御ルール
- 照度 < 200 lux → PWM LED照明ON（明るさを照度に反比例）
- 温度 > 26°C → ファン(モーター)ON（速度を温度に比例）
- 温度 < 18°C → LED赤で暖房警告
- 湿度 > 85% → ブザーで除湿アラート
- 距離 < 30cm → 人感検知 → 全照明MAX＋暖色ムードライト

### 結果（30サイクル）
- サイクル1-13: 距離 < 30cm → **人感検知モード** → 全灯ON
- サイクル14以降: 距離 > 30cm → 照度十分(>200lux) → **照明OFF**
- 温度: 20.2〜22.2°C → 空調不要範囲内
- 人感検知→離脱の遷移が正しく動作

### 知見
- **PWM LED調光**: `room_light.value = brightness` で0.0〜1.0の連続制御が可能
- **RGB LED暖色**: `(1.0, 0.9, 0.7)` で自然な暖色を表現
- **条件の優先順位**: 人感検知を最優先にすることで、他の条件との干渉を防止
- 距離センサーのこぎり波パターンにより、自然な「人が近づいて離れる」動作が再現された

## 実験8: MQTTエコシステム (`demo_mqtt_ecosystem.py`)

### 目的
3つの独立した仮想IoTノードがMQTT経由で協調動作するエコシステムの検証。

### アーキテクチャ
```
WeatherStation ──publish──> iot/weather/data
                                    │
GardenController <──subscribe───────┘
       │
       └──subscribe──> iot/security/alert ──> SecuritySystem (publish)
```

### ノード構成
| ノード | 役割 | MQTTトピック | デバイス |
|--------|------|-------------|---------|
| WeatherStation | 気象データ配信 | iot/weather/data (pub) | 温度/湿度/気圧/照度センサー |
| GardenController | 灌漑・日除け制御 | iot/garden/status (pub), iot/weather/data + iot/security/alert (sub) | Motor, Servo, 土壌水分 |
| SecuritySystem | 侵入検知・警報 | iot/security/alert (pub) | Buzzer, LED, 距離センサー |

### 結果（15サイクル）
- サイクル1-12: 距離 < 40cm → **侵入検知** → 警報12回
- SecuritySystem → `iot/security/alert` に `{"alert": true}` を配信
- GardenController → 警報受信 → **ポンプ強制停止** → ALERT MODE
- サイクル13: 距離 > 40cm → **警報解除**
- サイクル13-15: GardenController が通常動作に復帰

### 知見
- **MQTT Pub/Sub による疎結合**: 各ノードは相手の実装を知らなくても連携可能
- **リアルタイム反応**: 警報の publish → subscribe → 行動変更がほぼ即座
- **スレッド間でのMQTTクライアント分離**: 各ノードが独自の `mqtt.Client` を持つことで、subscription の干渉を防止
- **alert_mode パターン**: 警報受信時にフラグを立てて全制御を停止するパターンは、安全系IoTシステムの基本設計として有効
- **課題**: 距離センサーのこぎり波が起動直後は必ず小さい値から始まるため、テスト開始直後に必ず侵入検知になる。実システムでは起動安定待ち時間を設けるべき

## 実験9: データロガー + 統計分析 (`demo_data_logger.py`)

### 目的
IoTデータ収集パイプラインのプロトタイプ。50サンプルを記録し、統計分析と異常検出を行う。

### 手法
1. 6種センサーから50回サンプリング（0.1秒間隔）
2. CSV ファイルに保存
3. 各センサーの統計情報（平均/最小/最大/標準偏差/レンジ）を算出
4. 平均±2σ 逸脱をアノマリーとして検出
5. サマリーレポートを MQTT 配信

### 統計結果

| センサー | 平均 | 標準偏差 | 最小 | 最大 | レンジ |
|---------|------|---------|------|------|--------|
| 温度 (°C) | 22.38 | 0.75 | 21.26 | 23.82 | 2.56 |
| 湿度 (%) | 57.80 | 3.00 | 53.18 | 64.13 | 10.95 |
| 気圧 (hPa) | 1013.34 | 0.55 | 1011.96 | 1014.44 | 2.48 |
| 照度 (lux) | 512.42 | 20.82 | 463.10 | 552.26 | 89.16 |
| 距離 (cm) | 19.12 | 9.32 | 2.00 | 35.50 | 33.50 |
| 土壌水分 (%) | 68.65 | 0.76 | 67.34 | 69.90 | 2.56 |

### 異常検出結果（平均±2σ）
- **湿度**: 1件 (Sample #12: 64.13% HIGH)
- **気圧**: 3件 (Sample #17,18,39: LOW)
- **照度**: 1件 (Sample #18: 463.10 lux LOW)
- **合計**: 5件 / 300データポイント = 1.67% (統計的に期待される約5%以下、正常)

### 知見
- **CSVログ**: Pi VM 内に `/opt/iot-app/sensor_log.csv` として保存。SCPで取り出し可能
- **ランダムウォークの特性**: 温度・湿度はガウシアンノイズによるドリフト、気圧・照度はサイン波+ノイズ。パターンの違いが統計に反映される
- **距離の高レンジ**: のこぎり波パターンのため標準偏差が大きく（9.32cm）、他のセンサーとは分布特性が異なる
- **異常検出の有効性**: 2σルールで5件検出は統計的に妥当（正規分布なら約5%が範囲外）
- **MQTT配信**: レポートJSON は他のシステム（ダッシュボード、アラートシステム等）が subscribe して利用可能

## SSH経由でのプログラミングに関する知見

### 制約
- **heredoc 内のクオート**: SSH経由でヒアドキュメント内にシングルクオート、ダブルクオート、f-stringが混在するとエスケープが複雑化して破綻する
- **日本語の扱い**: f-string内の日本語リテラルもSSH経由ではエンコーディング問題が発生しやすい

### 推奨ワークフロー
1. **Windowsローカルでファイルを作成**（エディタ/IDE使用）
2. **SCP で Pi VM に転送**: `scp -P 2222 file.py root@localhost:/opt/iot-app/`
3. **SSH で実行**: `ssh -p 2222 root@localhost "cd /opt/iot-app && ./venv/bin/python file.py"`

このワークフローにより、複雑なエスケープ問題を完全に回避できる。
